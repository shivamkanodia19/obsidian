# ClinicalHours Next Wedge Priority - 2026-05-10

**Core recommendation:** For the current pilot and validation cycle, keep the broader ClinicalHours direction pointed toward small-clinic operations support, and treat `evaluation automation` as the best current hypothesis to test next, not yet a proven priority over onboarding or other admin workflows.

That means:

- build one narrow evaluation-automation MVP first
- treat onboarding reminders and status tracking as the next adjacent wedge
- keep broader workflow-platform language as a future direction, not a current proof claim
- keep pricing as a parallel business-model thread, not the main product-learning milestone

Pilot evidence supports testing evaluation automation first, but broader small-clinic demand for this wedge is still unproven. The other notes help with ideal-customer framing and messaging, not cross-clinic validation.

## Why This Is The Best Current Hypothesis To Test Next

The vault already points to the same pattern from multiple angles:

- [[02_Analyst/projects/ClinicalHours/clinicalhours-experiment-log-framing-2026-05-07]] says the clearest validated workflow is evaluation follow-up, not a broad market-wide platform claim
- [[niche-refinement]] keeps the focus on smaller clinics with manual or fragmented processes rather than enterprise-scale software complexity
- [[02_Analyst/projects/ClinicalHours/Outreach/DFW-Strategy]] helps frame clinic pain and outreach language, but it should be treated as persona and messaging support rather than hard validation evidence

Evaluation automation is the best current hypothesis to test next because it is:

- concrete enough to prototype quickly
- close to the pain already surfaced by the pilot
- easy to explain in a before-and-after workflow
- broad enough to hint at a larger operations layer without overclaiming that layer is already proven

## What To Build First

The first version should stay narrow:

1. identify which volunteers need post-session evaluation follow-up
2. trigger the right evaluation request automatically
3. store responses in the correct profile or record
4. let clinic staff review completion status and responses without manual email and Drive chasing

This is a better first proof point than a generic "workflow automation" pitch because it turns one real staff headache into one testable product behavior.

## What Comes Second

If the evaluation MVP works in practice, the next adjacent wedge should be:

- onboarding reminders
- deadline tracking
- status visibility across administrative steps

This is the clean expansion path because it stays inside the same underlying pain cluster: fragmented admin coordination across email, forms, Drive, and ad hoc follow-up.

## What Not To Do Yet

Do not do these things in the next round of strategy language:

- do not say the pilot proved a full clinic-operations platform
- do not call this a major pivot
- do not make pricing the headline learning from this stage
- do not jump straight to a hospital-tier story before the small-clinic workflow is tighter

## Practical 60-90 Day Sequence

### Phase 1: Validate The Narrow Workflow

- build the smallest useful evaluation-automation MVP
- run it with the pilot clinic
- measure whether it reduces manual follow-up work and coordination friction

### Phase 2: Expand Along The Same Pain Cluster

- add onboarding reminders and simple status tracking only if the first workflow gets used
- package the result as "small-clinic admin workflow support," not as a universal platform

### Phase 3: Tighten Positioning And Sales Language

- update demos and outreach around the strongest validated workflow
- use the broader automation story only as directionally true future positioning
- keep asking whether the clinic is buying software, operational relief, or both

## Messaging Implication

The strongest near-term story is:

"ClinicalHours helps small clinics reduce admin coordination work in high-friction workflows, starting with evaluation follow-up and onboarding."

That is stronger than:

- "ClinicalHours is a full clinic workflow platform"
- "The pilot proved broad workforce automation demand"

## Main Risk

The main risk is still overgeneralization from a small pilot footprint. Even if the evaluation workflow works well in one clinic, that does not yet prove the same urgency or willingness to adopt across the broader market.

## Next Actions

- prototype the narrow evaluation-automation MVP
- define 2-3 success signals before testing it
- treat onboarding reminders as the next wedge only if the first workflow gets real usage
- keep pricing work parallel and lightweight until product proof is stronger
