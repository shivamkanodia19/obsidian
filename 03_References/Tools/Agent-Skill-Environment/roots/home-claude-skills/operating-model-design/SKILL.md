# Operating Model Design

## When To Use

Use this skill when a strategy is clear but the organization needs to know how work should happen. It fits transformations, growth plays, new business units, functional redesigns, and scale-up challenges.

## McKinsey-Style Approach

Start with the strategy and required capabilities. Then design decision rights, roles, governance, processes, metrics, and enabling systems around those capabilities.

## Workflow

1. Define the strategy the operating model must support.
2. Identify required capabilities.
3. Diagnose current operating model gaps.
4. Design target roles, decision rights, governance, processes, and metrics.
5. Identify transition risks and capability build needs.
6. Recommend implementation phases.

## Output Format

```markdown
# Operating Model Design

## Strategic Requirement
[What the model must enable.]

## Capability Model
| Capability | Current State | Required State | Gap |
|---|---|---|---|

## Target Model
| Element | Recommendation | Rationale |
|---|---|---|

## Decision Rights
| Decision | Owner | Contributors | Governance |
|---|---|---|---|

## Transition Plan
[Phases, risks, and first moves.]
```

## Quality Bar

- Link every design choice to strategy.
- Make decision rights explicit.
- Include governance and metrics, not only org boxes.
- Show transition risks and sequencing.
